1. สรุปสถานะ

ระบบที่ implement แล้วคือ workflow นำเข้าทะเบียนตั้งต้น พร้อมหน้า public data-quality dashboard และหน้าเจ้าหน้าที่สำหรับ stage, approve และ confirm
ยังไม่ถือว่าพร้อม deploy รอบใหม่: วันที่ตรวจล่าสุด npm test และ npm run build รันไม่สำเร็จ เพราะ node_modules ไม่สมบูรณ์และหา executable ของ Vitest กับแพ็กเกจ esbuild ไม่พบ
โฟลเดอร์ dist/ มี artifact เดิมครบ 4 ไฟล์จากวันที่ 2 มิถุนายน 2026 แต่ dist/appsscript.json ปัจจุบันเป็น manifest ฝั่ง Staff จึงห้ามนำไป push ฝั่ง Public โดยไม่ build ใหม่

2. เทคโนโลยีและสถาปัตยกรรม

Application

  • TypeScript แบบ strict
  • Bundle ด้วย esbuild เป็น dist/Code.js
  • รันบน Google Apps Script V8
  • หน้าเว็บเป็น HTML/CSS/JavaScript และเรียก server ด้วย google.script.run

Data store

  • Google Sheets เป็นฐานข้อมูลกลาง
  • เปิดด้วย Spreadsheet ID ไม่พึ่ง active spreadsheet
  • ค่าเริ่มต้นอยู่ใน DEFAULT_DASHBOARD_SPREADSHEET_ID
  • เปลี่ยนได้ด้วย Script Property DASHBOARD_VACCINE_SPREADSHEET_ID

4. Workflow ทะเบียนตั้งต้นที่ถูกต้อง

1. Stage + Validate DISTRICT_APPROVER วาง TSV ระบบตรวจข้อมูล หากผ่านจึงบันทึกแถวใน BASELINE_STAGING และสร้าง batch สถานะ VALIDATED
2. Reject เมื่อมีปัญหา หากพบ issue ระบบสร้าง batch สถานะ REJECTED แต่ไม่บันทึกข้อมูลเด็กลง staging
3. Approve + Promote DISTRICT_APPROVER อนุมัติ batch จากนั้นระบบคัดลอกข้อมูลจาก staging เข้า CHILD_REGISTRY และนับเป็น provisional
4. Confirm UNIT_CONFIRMER ของหน่วยนั้นยืนยัน batch ให้เป็น UNIT_CONFIRMED และยอดจึงเปลี่ยนจาก provisional เป็น confirmed
จุดสำคัญ: ข้อมูลเข้า CHILD_REGISTRY ในขั้น Approve ไม่ใช่ขั้น Confirm ส่วน Confirm เป็นการรับรองฐานและเปลี่ยนสถานะการนับ coverage

การ Stage และ Approve ใช้ LockService.getDocumentLock() เพื่อป้องกันการเขียนพร้อมกัน หาก promote เข้า registry ล้มเหลว ระบบพยายามคืน batch กลับเป็นสถานะก่อนอนุมัติ

5. การตรวจข้อมูลนำเข้า

โครงสร้างไฟล์

  • หัวตารางต้องตรงกับ BASELINE_HEADERS ทั้ง 17 คอลัมน์และเรียงลำดับตรงกัน
  • หนึ่ง batch ต้องเป็นข้อมูลของหน่วยบริการเดียว
  • หน่วยบริการต้องอยู่ใน CFG_SERVICE_UNITS และ active

กฎระดับแถว

  • CID เป็นตัวเลข 13 หลัก ไม่ซ้ำในชุด และไม่ซ้ำกับ batch ที่ระบบยอมรับแล้ว
  • วันเกิดเป็นวันที่จริงรูปแบบ YYYY-MM-DD
  • สถานะวัคซีนอยู่ในรายการมาตรฐาน 5 ค่า
  • สถานะได้รับบางส่วนแต่ยังไม่ครบต้องมีวันนัดวัคซีนครั้งถัดไป
  • ต้องมีบ้านเลขที่ หมู่ สถานะทะเบียน ประเภทการเข้า เดือนเริ่มตัวชี้วัด อสม.หลัก และ ผรส.หลัก

6. Roles ที่ implement แล้ว

Roleทำได้ขอบเขตข้อมูล
DISTRICT_APPROVER Stage, Approve และดู batch ทุกหน่วย ระดับอำเภอ
UNIT_CONFIRMER Confirm และดู batch เฉพาะ service unit ของตน

Staff endpoint อ่านอีเมลจาก Session.getActiveUser().getEmail() และค้น role จาก CFG_BASELINE_USERS เฉพาะแถวที่ active หากไม่มีบัญชี พบซ้ำ หรือ role ไม่ถูกต้อง ระบบจะปฏิเสธการทำงาน

7. Google Sheets ที่ระบบสร้าง

Sheetหน้าที่ผู้ดูแลข้อมูล
CFG_SERVICE_UNITSรหัส ชื่อ และสถานะ active ของหน่วยบริการต้องกรอก configuration จริง
CFG_BASELINE_USERSอีเมล role หน่วยบริการ และ active ของผู้ใช้ต้องกรอก configuration จริง
BASELINE_BATCHESสถานะ batch, จำนวนแถว, issues, ผู้ทำรายการ และเวลาระบบ
BASELINE_STAGINGข้อมูลดิบของ batch ที่ผ่าน validation รออนุมัติระบบ
CHILD_REGISTRYข้อมูลจาก batch ที่ได้รับอนุมัติระดับอำเภอแล้วระบบ

10. Build, Test และ Deploy

คำสั่งผลที่ควรเกิดสถานะตรวจล่าสุด
npm testรัน Vitest ทั้งหมดรันไม่ได้: ไม่พบ vitest executable
npm run buildสร้าง public bundle และคัดลอก public manifestรันไม่ได้: ไม่พบ esbuild package
npm run build:staffสร้าง bundle และคัดลอก staff manifestยังยืนยันไม่ได้ด้วยสาเหตุเดียวกัน
npm run push:publicBuild แล้ว push ไป public Script IDยังไม่ควรรันจน test/build ผ่าน
npm run push:staffBuild staff แล้ว push ไป staff Script IDยังไม่ควรรันจน test/build ผ่าน

ไฟล์ .clasp.public.json และ .clasp.staff.json มี Script ID ตั้งค่าอยู่แล้ว แต่การมี target ไม่ได้แปลว่า deployment ล่าสุดถูกสร้างหรือทำงานผ่านการตรวจสอบแล้ว

11. สิ่งที่ยังไม่มีใน implementation ปัจจุบัน

Core monthly workflow

  • สถานะวัคซีนรายเดือนและ carry-forward
  • pending verification และการรับรองสถานะทางการ
  • submission, district close และการล็อกรอบ
  • KPI จากหน่วยที่ยืนยันพร้อม coverage ที่ชัดเจน
  • audit log สำหรับการแก้ย้อนหลัง

Supplementary modules

  • บัญชีและ mobile follow-up ของ อสม./ผรส.
  • กิจกรรมบริการเชิงรุก
  • funnel วัคซีนทางเลือกและ export aggregate
  • notification และ escalation
  • backup, restore, outage และ security incident workflow

หัวข้อเหล่านี้มีการตัดสินใจและออกแบบไว้ใน CONTEXT.md และ docs/adr/ แต่ไม่ควรถูกอธิบายว่าเป็นฟีเจอร์ที่ใช้งานได้แล้ว
