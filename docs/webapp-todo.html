Todo - Dashboard Vaccine

รายการงานจากสถานะ repository ณ วันที่ 12 มิถุนายน 2026 เรียงจากสิ่งที่ต้องทำก่อน deploy ไปยัง roadmap ระยะถัดไป

กลับไปหน้า Overview
P0ซ่อมและยืนยัน build ก่อน deploy
P1ตั้งค่าและ dry run ระบบ baseline
P2+พัฒนา monthly workflow และโมดูลถัดไป

P0 - ต้องเสร็จก่อน Deploy รอบใหม่

ติดตั้ง dependencies ใหม่ให้สมบูรณ์

โฟลเดอร์ node_modules ปัจจุบันขาด Vitest executable และ package metadata ของ esbuild ควรติดตั้งจาก package-lock.json แล้วตรวจว่า node_modules/.bin ถูกสร้างครบ

P0

รันชุดทดสอบทั้งหมด

รัน npm test และแก้เฉพาะความล้มเหลวจริงจนทุก test ผ่าน รวม validation, access control, repository, workflow, routing และ privacy guard

P0

Build Public และตรวจ manifest

รัน npm run build แล้วตรวจว่า dist/appsscript.json เป็น USER_DEPLOYING / ANYONE_ANONYMOUS และมี artifact ครบ 4 ไฟล์

P0

Build Staff และตรวจ manifest

รัน npm run build:staff แล้วตรวจว่า dist/appsscript.json เป็น USER_ACCESSING / DOMAIN ก่อน push ไป Staff project

P0

ยืนยัน privacy ของ Public endpoint

ตรวจผลจาก getPublicDashboardModel() ว่าไม่มี CID, ชื่อเด็ก, วันเกิด, ที่อยู่, ผู้ปฏิบัติงาน, raw record, issue รายแถว หรือข้อมูล audit

P0

P1 - ตั้งค่าระบบและ Dry Run

ยืนยัน Spreadsheet กลางและสิทธิ์เข้าถึง

ตรวจ Spreadsheet ID ที่ใช้จริง ตั้ง Script Property ในทั้งสอง project หากไม่ใช้ค่า default และยืนยันว่าผู้ deploy กับ staff users มีสิทธิ์ตามรูปแบบ execute-as

P1

Provision ตาราง baseline

รัน provisionBaselineTables() ครั้งแรก แล้วตรวจหัวตารางทั้ง CFG_SERVICE_UNITS, CFG_BASELINE_USERS, BASELINE_BATCHES, BASELINE_STAGING และ CHILD_REGISTRY

P1

กรอกหน่วยบริการ 14 แห่ง

กรอกรหัส 5 หลัก ชื่อทางการ และ active = TRUE ใน CFG_SERVICE_UNITS พร้อมตรวจชื่อและรหัสกับรายการที่อนุมัติ

P1

กำหนดผู้ใช้ baseline จริง

กรอกอีเมล DISTRICT_APPROVER และ UNIT_CONFIRMER พร้อม service-unit scope ใน CFG_BASELINE_USERS และตรวจว่าไม่มีอีเมล active ซ้ำ

P1

ทำ dry run หนึ่งหน่วยบริการ

ทดสอบข้อมูลจริงตาม docs/runbooks/baseline-dry-run.md: invalid data ต้องถูก reject, VALIDATED ยังไม่เข้า registry, Approve ต้อง promote เข้า registry, Confirm ต้องเปลี่ยน coverage เป็น 1 จาก 14

P1

ตรวจสิทธิ์ด้วยบัญชีจริง

ยืนยันว่า district approver มองเห็นทุก batch, unit confirmer เห็นเฉพาะหน่วยตน, wrong-unit confirmer ถูกปฏิเสธ และ public URL ไม่สามารถใช้ทำ staff action ได้

P1

Deploy และบันทึก Web App URLs

หลัง verification ผ่าน ให้ push และสร้าง deployment แยก Public/Staff จากนั้นบันทึก URL เวอร์ชันที่ใช้งานจริงและทดสอบจากอุปกรณ์ desktop/mobile

P1
เกณฑ์ผ่าน P1: import ที่ถูกต้องทำงานครบทั้ง Stage → Approve/Registry → Confirm, coverage ถูกต้อง, role scope ถูกต้อง, Public ไม่มีข้อมูลรายบุคคล และมีหลักฐาน actor/timestamp ใน batch

P2 - Core Monthly Workflow

ออกแบบและสร้าง schema รายเดือน

เพิ่มโครงสร้างสำหรับสถานะวัคซีนรายเดือน, reporting month, carry-forward และ monthly confirmation โดยยึด ADR 0015, 0018 และ 0019

P2

สร้าง pending verification workflow

แยกข้อค้นพบภาคสนามออกจากสถานะทางการ และให้หน่วยบริการรับรองก่อนเปลี่ยน official vaccine status

P2

Submission, close และ lock รอบเดือน

รองรับการยืนยันส่งข้อมูลรายหน่วย การปิดรายงานระดับอำเภอแม้ส่งไม่ครบ การแสดง incompleteness และการควบคุมแก้ย้อนหลัง

P2

Audit log แกนหลัก

บันทึกผู้กระทำ เวลา ค่าเดิม ค่าใหม่ เหตุผล และการอนุมัติสำหรับการเปลี่ยนสถานะหรือแก้ข้อมูลย้อนหลัง

P2

เปิดข้อมูลจริงใน Vaccine Progress dashboard

เมื่อ monthly aggregate backend พร้อม จึงเปลี่ยนจาก NOT_READY เป็นข้อมูลสรุประดับอำเภอ/หน่วยบริการและ movement ระหว่างเดือน โดยไม่เปิดข้อมูลรายเด็ก

P2

P3 - โมดูลปฏิบัติการเพิ่มเติม

Field-worker mobile follow-up

บัญชีรายบุคคล, session จำกัดเวลา, ข้อมูลเด็กเท่าที่จำเป็น, controlled reasons, edit window และ central event log

P3

กิจกรรมบริการเชิงรุก

กิจกรรมตามแผน, ผู้รับผิดชอบ, กำหนดปิดกิจกรรม, ข้อยกเว้น และผลลัพธ์ aggregate

P3

วัคซีนทางเลือก

บันทึก consent status และ service decision โดยไม่เก็บเอกสารต้นฉบับ พร้อม funnel aggregate และ export สำหรับ อบจ.

P3

Notification และ escalation

แจ้งเตือนงานค้างระดับหน่วยและยกระดับไปยังอำเภอตาม ADR 0040

P3

P4 - ความมั่นคงและการดำเนินงานระยะยาว

Backup และ restore

กำหนด backup point ก่อนเปลี่ยนโครงสร้างและก่อนปิดเดือน ทดสอบกู้คืน และบันทึกหลักฐาน restoration ตาม ADR 0043

P4

Offline/outage fallback

รองรับการบันทึกสำรองในช่วงระบบล่ม พร้อมการนำกลับเข้า การตรวจซ้ำ และ audit โดยไม่ทำให้รายงานดูครบเกินจริง

P4

Security incident workflow

กำหนดการบันทึก ควบคุมผลกระทบ เปลี่ยนสิทธิ์ ตรวจสอบ และปิด incident ตาม ADR 0041

P4

Project close และ retention

กำหนดสถานะหลังสิ้นสุดโครงการ การปิดการแก้ไข การเก็บรักษาข้อมูล และผู้รับผิดชอบระบบระยะยาว

P4
Todo นี้เป็นเอกสารสถานะ ไม่ได้เชื่อมกับ issue tracker และ checkbox ไม่บันทึกค่าเมื่อเปิดใหม่
